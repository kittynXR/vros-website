Service-based architecture

Director + services + UIs, connected over local IPC.

Discovery and shutdown

Services register to the director on a control channel. The director assigns channel names and coordinates shutdown.

Named pipes and shared memory

Control/command/event channels default to named pipes; data channels can use shared memory for throughput.

Desktop + in-VR overlays

Operate vrOS from a desktop control center and in-VR overlays (dashboard/tools/capture/keyboard).

Built for real workflows

Streaming tools, automation, and overlay control, backed by a service runtime.

Capture overlay

Capture windows and displays into VR with configurable throttling and interaction/pointer forwarding.

In-VR keyboard

Keyboard overlay with routing support, layout/theme configuration, and VR-friendly focus behavior.

Chat and integrations

Unified chat surface integrating Twitch chat and VRChat OSC, with settings persistence.

OBS control

Connect to OBS and control scenes/sources and stream/record workflows through the OBS service.

Scripting

Script library + editor, run/stop, output console, and an API browser for automation workflows.

Audio control

Per-application volume/mute controls, device/session lists, and live audio level metering.

How it connects

Service discovery and channel assignment are explicit, not magic.

Channel types

Command / Event Named Pipe
Data Shared Memory
Negotiation Bootstrap SHM

Director-assigned channels

Services announce themselves; the director replies with the channel names and metadata to use for connections.

Crash isolation

Work is split across processes/services so failures are contained and UIs can degrade gracefully.

Configuration-driven

Services and overlays read config files at startup to decide what channels to open and how to behave.
